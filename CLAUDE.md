## Core JS rule

**Each page loads only the JS it needs.**

The centralised router in `layout/theme.liquid` uses `request.page_type` to include
the correct entrypoint. Never add `{% render 'vite-tag' %}` inside section files —
loading is handled exclusively by the layout.

| Always loaded | Template-specific only |
|---|---|
| `css/main.css` | `ts/index.ts` (homepage) |
| `ts/theme.ts` | `ts/product.ts` (PDP) |
| | `ts/collection.ts` (PLP) |
| | `ts/cart.ts`, `ts/search.ts`, etc. |

`ts/gift-card.ts` is the only exception: loaded directly in `templates/gift_card.liquid`
because that template uses `{% layout none %}`.

## When to add JS

- **Global bootstrap only**: `frontend/entrypoints/ts/theme.ts`
- **Shared logic/utilities** (e.g. modal, DOM helpers): `frontend/entrypoints/ts/utils/*`
- **Template-specific logic**: the matching entrypoint (e.g. `frontend/entrypoints/ts/product.ts`)
- **Never create a new entrypoint** without also updating the router in `layout/theme.liquid`

## Theme preview

### Prerequisites

```bash
bun install  # or npm install
cp .env.example .env
cp example.shopify.theme.toml shopify.theme.toml
```

### Store configuration

- `.env` controls Vite/preview origin settings
  - `SHOPIFY_STORE_DOMAIN=your-store.myshopify.com`
- `shopify.theme.toml` controls Shopify CLI environment
  - `[environments.development]`
- Keep `shopify.theme.toml` local (gitignored) and commit only `example.shopify.theme.toml`

### Local dev (recommended for daily development)

```bash
bun run dev
```

- Shopify CLI at `http://127.0.0.1:9292`
- Vite at `http://localhost:5173` (HMR active)
- CSS/JS served directly from the Vite dev server

### Remote dev (for preview/editor on the Shopify domain)

```bash
bun run dev:remote
```

- Same as local but with Cloudflare tunnel enabled via `vite-plugin-shopify`
- Required when opening the preview at `https://${SHOPIFY_STORE_DOMAIN}`
- Chrome blocks requests from a public origin to `localhost` (PNA policy);
  the tunnel fixes this by exposing Vite over a public HTTPS URL

### Deploy

```bash
bun run build
```

Builds production assets for branch-linked Shopify themes.

### Branch-linked Shopify deploy note

If your Shopify theme is connected directly to a Git branch (no `shopify theme push`), you must commit built assets to the branch.

Required flow:
1. Run `bun run build`
2. Commit generated files (`assets/*`, `assets/.vite/manifest.json`, `snippets/vite-tag.liquid`)
3. Push the branch

Important:
- CI checks/builds do not automatically update your branch unless the workflow commits artifacts.
- Shopify does not run `vite build` for branch-linked themes.

## Git and branch strategy

### Branch model

- `main`: production-ready branch (connected to the production Shopify theme)
- `staging`: QA/integration branch (connected to a staging Shopify theme)
- `feat/*`: feature branches (isolated development work)

### Merge flow

1. Build on `feat/<feature-name>`
2. Open PR to `staging`
3. QA on the Shopify theme linked to `staging`
4. Merge `staging` into `main` for release
5. Keep `main` always releasable

### Branch-linked deploy rule

If a Shopify theme is connected to a Git branch, you must commit built assets on that branch.

Required on `staging` and `main`:
1. Run `bun run build`
2. Commit generated files (`assets/*`, `assets/.vite/manifest.json`, `snippets/vite-tag.liquid`)
3. Push the branch

CI checks/builds alone do not update branch-linked Shopify themes unless the workflow commits artifacts.

### PR quality gates

Before merging to `staging` or `main`, ensure:
- `bun run typecheck` passes
- `bun run build` passes
- `theme-check` passes
- Smoke test the templates touched by the PR

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| CSS not loading | Vite not running | `bun run dev` |
| CSS not loading on `myshopify.com` | Chrome PNA block (loopback) | `bun run dev:remote` |
| Port 5173 already in use | Stale Vite process | `lsof -nP -iTCP:5173 -sTCP:LISTEN` then kill PID |
| Port 9292 already in use | Stale Shopify CLI process | `lsof -nP -iTCP:9292 -sTCP:LISTEN` then kill PID |
| CORS error on `/api/collect` | Shopify analytics on local origin | Non-blocking warning, safe to ignore in dev |

## Working with Claude efficiently

### Reduce token usage

- **Always name the file first**: "edit `sections/product.liquid`" not "edit the product page"
- **Reference line numbers** when possible: "line 15 in `layout/theme.liquid`"
- **One task at a time**: confirm before moving to the next step
- **Skip general Shopify/Vite explanations** unless strictly needed

### Recommended task format

1. Target file (`sections/collection.liquid`)
2. Goal (what it should do)
3. Approve the plan before execution

### Naming conventions

- TS entrypoints: match the Shopify `request.page_type` value
- Sections: `{page_type}.liquid` (e.g. `product.liquid`, `blog.liquid`)
- Snippets: descriptive kebab-case name
